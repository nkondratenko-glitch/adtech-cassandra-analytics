## Kondratenko | HW4 | Advertising Event Data in Apache Cassandra

This homework models high-velocity advertising event data in Apache Cassandra. Instead of normalizing data like in MySQL, Cassandra tables are designed per query so reads stay fast for the exact access patterns required by the business.

## Why Cassandra

Cassandra is a good fit for:
- time-series metrics such as impressions and clicks per day
- high write throughput for ad events
- fast retrieval by partition key
- denormalized, query-oriented storage

## Query-driven table design

This homework uses separate tables for different read patterns:

### 1. `campaign_daily_metrics_snapshot_by_campaign`
Stores daily campaign metrics by `campaign_id`, including impressions, clicks, spend, and precomputed CTR.
Used for:
- ad campaign performance over time
- CTR per campaign per day

### 2. `user_ad_history_by_user`
Stores user impression history ordered by most recent event time.
Used for:
- last 10 ads seen by a user
- whether the user clicked each ad

### 3. `advertiser_spend_by_day`
Stores daily advertiser spend by day and region.
Used as a base table for snapshot rollups.

### 4. `advertiser_spend_30d_snapshot`
Stores precomputed 30-day spend rankings for advertisers.
Used for:
- top advertisers by total ad spend in the past 30 days

### 5. `user_clicks_30d_snapshot`
Stores precomputed 30-day click rankings for users.
Used for:
- top 10 most active users by clicks in the last 30 days

### 6. `advertiser_spend_30d_by_region_snapshot`
Stores precomputed 30-day regional advertiser rankings.
Used for:
- top spending advertisers by region over the last 30 days

## Files included

- `scripts/10_load_cassandra_data.sh` – loads data into Cassandra
- `scripts/11_run_demo_queries.sh` – runs sample queries through Python
- `requirements_hw4.txt` – Python dependencies

## Important modeling note

Cassandra does not handle global sorting and cross-partition aggregations the way relational databases do. For ranking questions such as "top 5 advertisers in the last 30 days", this homework uses precomputed snapshot tables generated by the ETL script.

This is a normal Cassandra pattern: write data into tables that directly match the queries you need.

## How to run

1. Start Cassandra:

```bash
docker compose -f docker-compose.cassandra.yml up -d
```

2. Create schema:

```bash
docker exec -i adtech-cassandra cqlsh < cassandra/schema.cql
```

3. Install dependencies:

```bash
pip install -r requirements_hw4.txt
```

4. Load data from MySQL:

```bash
python scripts/load_mysql_to_cassandra.py
```

5. Run query export:

```bash
python scripts/run_cassandra_queries.py
```

## Assumptions

The loader expects the Homework 1 MySQL schema with tables such as:
- `advertisers`
- `campaigns`
- `users`
- `locations`
- `impressions`
- `clicks`

If `impressions` and `clicks` are sparse or empty, some Cassandra tables may contain little or no event data. The schema and ETL still remain correct.
